# Risky Entra sign-in followed by mailbox-rule mutation (Sigma)

- **Query file:** `T1078.004_risky-signin-mailbox-rule.yml`
- **Author:** nextlayersec
- **Last updated:** 2026-06-04
- **Status:** experimental
- **Severity:** High (correlation); base rules are Medium and Low.

## What it detects

Vendor-neutral counterpart to the KQL rule [`T1078.004_risky-signin-mailbox-rule.kql`](../kql/T1078.004_risky-signin-mailbox-rule.md). The same UPN appears in both a risky successful Entra sign-in and an Exchange Online inbox-rule mutation within a 10-minute window — the canonical post-AiTM cookie-replay signal.

## ATT&CK mapping

- T1078.004 – Valid Accounts: Cloud Accounts
- T1564.008 – Email Hiding Rules
- Tags: `attack.initial_access`, `attack.t1078.004`, `attack.defense_evasion`, `attack.t1564.008`

## Data sources

Two cross-source base rules plus a temporal correlation:

| Doc | Logsource | Coverage |
|---|---|---|
| `entra_risky_successful_signin` | `{product: azure, service: signinlogs}` | Entra ID sign-in events with `ResultType: 0` and `RiskLevelDuringSignIn` / `RiskLevelAggregated` in `medium` / `high` |
| `entra_inbox_rule_mutation` | `{product: m365, service: exchange}` | Exchange UAL `New-InboxRule` / `Set-InboxRule` / `UpdateInboxRules` / `Enable-InboxRule` |
| Correlation | `type: temporal`, `group-by: UserPrincipalName`, `timespan: 10m`, `ordered: true` | Risky sign-in must precede the rule mutation by ≤ 10m |

`pysigma 1.3.3` validates the file (3 docs, including the correlation).

## SIEM-side aggregation note

Not all `sigma convert` backends support the `temporal` correlation type. Conversion guidance:

- **Sentinel (KQL):** prefer the [KQL sibling](../kql/T1078.004_risky-signin-mailbox-rule.md) — it joins `SigninLogs` + `OfficeActivity` natively with the same 10-minute window.
- **Splunk:** run each base rule as a saved search → use a `tstats` / `transaction` follow-up grouped by `UserPrincipalName` with `maxspan=10m`.
- **Elastic:** convert base rules to EQL `sequence by user.name with maxspan=10m [signin] [inbox_rule]`.

## False positives

- Power users restructuring folders right after a risky-rated travel sign-in.
- Service accounts owned by managed-mailbox tooling that templates rules at first sign-in.
- VPN egress that gets the original sign-in rated risky.

## Tuning

- Tighten by adding a positive-fidelity branch when the rule has `ForwardTo` / `RedirectTo` to an external domain — that combination is near-certain compromise; escalate to Critical and page.
- Add a negative filter for `Operation: Set-InboxRule` events with no `MoveToFolder` / `ForwardTo` / `RedirectTo` / `DeleteMessage` parameter set (pure flag changes are usually benign).
- Cross-reference the `SigninIP` against threat-intel; AiTM landing-page IPs and residential-proxy pools are high-leverage joins.

## Response

See the KQL sibling — same incident-response flow. Treat this as a page-the-on-call alert when the rule does anything destructive (forward / redirect external, delete, move to hidden folder).
