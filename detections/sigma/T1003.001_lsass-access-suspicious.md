# Suspicious LSASS Process Access (Sigma)

- **Query file:** `T1003.001_lsass-access-suspicious.yml`
- **Author:** nextlayersec
- **Last updated:** 2026-05-28
- **Status:** experimental
- **Severity:** High

## What it detects

Non-Microsoft, non-EDR processes that open a handle to `lsass.exe` with access rights that enable memory reading or full process control — the precondition for credential dumping (Mimikatz, ProcDump abuse, `comsvcs.dll MiniDump`, direct syscall variants).

## ATT&CK mapping

- **Tactic:** Credential Access (TA0006)
- **Technique:** T1003.001 – OS Credential Dumping: LSASS Memory
- **Related:** T1055 – Process Injection (often paired)

## Data sources

- Log source: `process_access` (Windows)
- Sysmon: Event ID 10 (`ProcessAccess`) — this rule's primary source; `GrantedAccess` and `SourceImage`/`TargetImage` come from EID 10.
- Defender XDR / MDE: `DeviceEvents` with `ActionType == "OpenProcessApiCall"`
- Sentinel: same `DeviceEvents` table via the M365D connector
- Minimum retention recommended: 30 days

## Logic

Matches Sysmon ProcessAccess events where `TargetImage` ends in `\lsass.exe` and `GrantedAccess` is one of the dump-capable masks (`0x1010`, `0x1410`, `0x1438`, `0x143A`, `0x1FFFFF`). The negative `filter_known` block subtracts Microsoft Defender / EDR binaries by `SourceImage`. Final condition: `selection and not filter_known`.

## False positives

- Endpoint protection products (CrowdStrike, SentinelOne, Carbon Black) — add their signer or image name to `filter_known`.
- Backup and DLP agents that snapshot process memory.
- Microsoft Process Explorer when run by admins for support.

## Tuning

- Prefer a signer allowlist over an image allowlist where your backend supports it (e.g. Microsoft-signed publisher) for fewer maintenance breakages.
- Anything launching from `\Users\*` or `\Windows\Temp\` is high-fidelity — escalate.
- Convert to your SIEM with `sigma convert -t <backend> detections/sigma/T1003.001_lsass-access-suspicious.yml`.

## Response

Containment: isolate host (EDR), force password reset for every user logged in within the previous 24h, rotate machine account / Kerberos TGT. Post-incident: hunt for resulting Pass-the-Hash / Kerberos abuse with sibling detections.
