# Run / RunOnce Key Persistence (Sigma)

- **Query file:** `T1547.001_runkey-persistence.yml`
- **Author:** nextlayersec
- **Last updated:** 2026-06-02
- **Status:** experimental
- **Severity:** Medium

## What it detects

Writes to Windows **Run**, **RunOnce**, **RunOnceEx**, or Winlogon `Shell` / `Userinit` registry values, excluding common installer / updater binaries via an `Image|endswith` allowlist. Vendor-neutral counterpart to the KQL rule [`T1547.001_runkey-persistence.kql`](../kql/T1547.001_runkey-persistence.md).

## ATT&CK mapping

- **Tactic:** Persistence (TA0003)
- **Technique:** T1547.001 – Boot or Logon Autostart Execution: Registry Run Keys / Startup Folder
- **Tags:** `attack.persistence`, `attack.t1547.001`

## Data sources

- Sysmon **Event ID 13** (RegistryValueSet) and **Event ID 12** (RegistryCreateKey/SetValue) on Windows endpoints.
- Equivalent Sigma `logsource`: `{product: windows, category: registry_event}`.
- For backends that don't surface Sysmon: any registry-write telemetry (EDR registry events, Windows Security 4657 if Object Access auditing is on).

## Logic

Two `selection_*` clauses combine with two `filter_*` clauses under `condition: selection_runkey and selection_writeop and not 1 of filter_*`:

- `selection_runkey` — `TargetObject` ends with one of the persistence-relevant key paths.
- `selection_writeop` — restricts to `SetValue` / `CreateKey` event types (drops deletes).
- `filter_installers` — common installer / updater image names.
- `filter_empty` — Sysmon emits `Details: (Empty)` on registry-key creations with no payload; those are not interesting on their own.

## False positives

- In-house deployment / packaging tools (Chocolatey, Intune scripts, custom MSI wrappers).
- Vendor agents (printer drivers, VPN clients, creative-tool helpers) that pin themselves into Run.
- Game launchers and auto-updaters not covered by the allowlist.

## Tuning

- Replace `filter_installers` with a signer-based filter when your backend supports `Signature` / `SignatureStatus` (e.g., Sysmon with the `ProcessAccess` signer fields, Splunk CIM `Authentication.signature`).
- Add a positive-fidelity branch for `Details|contains` on `powershell.exe`, `mshta.exe`, `rundll32.exe`, `wscript.exe`, `cscript.exe`, or `-enc` / `FromBase64String`.
- If you have HKCU vs HKLM context, split the rule — HKLM writes require local admin and are higher severity.

## Backend notes

- `sigma convert -t splunk T1547.001_runkey-persistence.yml` works out of the box.
- For Sentinel / Defender XDR, the canonical KQL implementation in [`detections/kql/T1547.001_runkey-persistence.kql`](../kql/T1547.001_runkey-persistence.md) uses `DeviceRegistryEvents` with the same filter set and adds an HKCU/HKLM flag — prefer that if you have Defender for Endpoint.
