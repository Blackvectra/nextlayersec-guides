# Delivered Phish: Attachment + Credential-Harvester Link (Sigma)

- **Rule file:** `T1566.001_attachment-link-credential-harvester.yml`
- **Author:** nextlayersec
- **Last updated:** 2026-06-04
- **Status:** experimental
- **Severity:** High (correlation) — base rules are informational/low on their own.

## What it detects

The vendor-neutral counterpart to the
[KQL detection](../kql/T1566.001_attachment-link-credential-harvester.md). It
fires when the **same message** is both:

1. a **delivered inbound** email carrying an **attachment**, and
2. carrying a **link to a credential harvester** on abused free / trusted-SaaS
   hosting (Google Sites, Firebase, Azure Blob, `*.pages.dev`, etc.).

That combination is the attachment-borne credential phish that bypassed
filtering — first observed in
**[INC-2026-001](https://github.com/Blackvectra/nextlayersec-incidents/tree/main/incidents/2026/INC-2026-001-bchabis-phishing)**
(`bchabis[.]com`: a password-protected PDF with a `sites.google.com/view`
harvester embedded as 39 link annotations).

## ATT&CK mapping

- **Tactic:** Initial Access / Execution
- **Primary technique:** T1566.001 – Phishing: Spearphishing Attachment
- **Co-fires with:** T1204.002 – User Execution: Malicious File; T1656 – Impersonation

## Structure

Three documents in one file (matches the repo's correlation-rule pattern):

| Rule | Type | Role |
|------|------|------|
| `m365_delivered_inbound_with_attachment` | base | Delivered inbound mail with ≥1 attachment |
| `m365_email_url_abused_hosting` | base | URL whose host is abused free hosting |
| *(correlation)* | `temporal` | Both, same `NetworkMessageId`, within 5m → High |

## Conversion

```bash
pip install "pysigma>=1.3,<2" sigma-cli
# example: Microsoft 365 Defender / Sentinel KQL backend
sigma convert -t kusto detections/sigma/T1566.001_attachment-link-credential-harvester.yml
```

Field names (`EmailDirection`, `DeliveryAction`, `AttachmentCount`, `UrlDomain`,
`NetworkMessageId`) follow the **Defender XDR advanced-hunting** schema. Adjust
field mappings if your backend uses a different taxonomy.

## Limitations vs. the KQL version

Sigma cannot cleanly express the JSON `AuthenticationDetails` parsing or the
attachment SHA-256 enrichment, so this rule is the **portable base**: it lacks
the weak-auth (`SPF=none` / `DMARC=bestguesspass`) scoring and the lookalike-
domain check. Use the
[KQL detection](../kql/T1566.001_attachment-link-credential-harvester.md) for
the higher-fidelity, fully-scored version and the clicker check.

## False positives

- Legitimate attachment mail that also links to a file on Google Drive / Firebase.
- Newsletters using free static hosting for landing pages.

Tune by pruning the `UrlDomain` list to platforms you do not use legitimately,
and by allowlisting known-good sender domains downstream.
