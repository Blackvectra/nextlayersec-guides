## What it detects

`rundll32.exe` invoked with one of the high-fidelity abuse patterns. The Sigma rule fires when **any** of five branches is true under a `rundll32` ancestor selection:

| Branch | What it matches | Why |
|---|---|---|
| `protocol_bridge` | `javascript:`, `vbscript:`, `data:`, `mshtml,RunHTMLApplication`, `url.dll,FileProtocolHandler`, `url.dll,OpenURL`, `shell32.dll,ShellExec_RunDLL http` substrings | Protocol-handler bridge ‚Äî the classic `rundll32 javascript:"\..\mshtml,RunHTMLApplication"` JScript loader and friends. Has no legitimate use. |
| `unc_or_http_dll` | `rundll32 \\‚Ä¶` or `rundll32 http(s)://‚Ä¶` (regex) | Windows happily loads DLLs from UNC / HTTP ‚Äî attacker-supplied DLLs land here. |
| `writable_path` | DLL path under `\Users\Public\`, `\AppData\*`, `\ProgramData\`, `\Windows\Temp\`, `\Temp\`, `%TEMP%`, `%TMP%` | World-writable / user-controllable drop path is a near-certain malware drop pattern. |
| `weird_extension` | First positional arg matches `.tmp / .dat / .log / .png / .jpg / .ico / .pdf / .txt / .zip / .gif / .bin / .rar` | Disguised payload extensions used by loaders to dodge naive binary-extension filters. |
| `no_args` | `rundll32` / `rundll32.exe` with no arguments | Canonical process-hollowing target. Rare in normal operation. |

## Data sources

- **Sigma logsource:** `product: windows`, `category: process_creation`.
- **Backends:**
  - **Defender XDR / Sentinel** ‚Äî convert with `sigma convert -t kusto`. The companion KQL (`detections/kql/T1218.011_rundll32-unusual-cmdline.kql`) is the hand-tuned native version with per-pattern boolean columns + scoring.
  - **Sysmon** ‚Äî use `sigma convert -t sysmon`. Maps to Event ID 1 (`ProcessCreate`).
  - **Splunk** ‚Äî `sigma convert -t splunk`. Use against `index=wineventlog EventCode=4688` or your Sysmon index.
  - **Elastic** ‚Äî `sigma convert -t elasticsearch`. Maps to `winlog.event_data.Image` / `winlog.event_data.CommandLine`.

## False positives

- Microsoft-signed UI plumbing ‚Äî `rundll32 shell32.dll,Control_RunDLL <path-to-cpl>` for a legitimate `.cpl` from `\Windows\System32\` launched by `control.exe` / `explorer.exe`. The Sigma rule does NOT have a `ParentImage` exclusion (Sigma rules are kept conservative); the KQL companion adds the parent allowlist explicitly.
- Printer / network panels ‚Äî `rundll32 printui.dll,PrintUIEntry` from `spoolsv.exe` / `explorer.exe`.
- Vendor agents (printer / VPN / endpoint) that register rundll32 entries in Run keys ‚Äî verify signer before allowlisting in your local fork.
- Software installers / removers ‚Äî `msiexec.exe`, legitimate `setup.exe` parents are common false-positive sources for the `writable_path` branch.

## Tuning

- **Add a `ParentImage` exclusion** in your local fork if your backend supports it: exclude `\explorer.exe`, `\svchost.exe`, `\services.exe`, `\msiexec.exe`, `\control.exe`, `\spoolsv.exe`, `\mmc.exe` from the `writable_path` and `weird_extension` branches only (leave `protocol_bridge`, `unc_or_http_dll`, `no_args` alone ‚Äî those are dangerous from any parent).
- **`writable_path` is the loudest branch.** Build a per-publisher allowlist (`DeviceFileCertificateInfo` join in KQL, `Signature.Subject` in Elastic ECS) if your fleet has Click-to-Run / browser-plugin software that legitimately ships rundll32 calls into `\AppData\`.
- **`no_args` precision** ‚Äî in some environments, AV scanners create no-arg rundll32 processes during sandbox inspection. Tighten to "no-args AND parent NOT in trusted-AV-list".
