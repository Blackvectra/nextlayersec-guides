# Entra ID Password Spray (Sigma)

- **Query file:** `T1110.003_entra-password-spray.yml`
- **Author:** nextlayersec
- **Last updated:** 2026-05-28
- **Status:** experimental
- **Severity:** High

## What it detects

A single source IP that produces failed sign-ins against many distinct user principals in a short window ‚ÄĒ the signature of a password spray. The rule ships as two YAML documents: a noisy base rule that matches credential-failure `ResultType` codes, and a correlation rule that aggregates those failures by source IP.

## ATT&CK mapping

- **Tactic:** Credential Access (TA0006)
- **Technique:** T1110.003 ‚Äď Brute Force: Password Spraying
- **Related:** T1078.004 ‚Äď Valid Accounts: Cloud Accounts (post-success follow-up)

## Data sources

- Log source: `azure` / `signinlogs`
- Entra connector: sign-in logs exported via diagnostic settings ‚Üí Log Analytics / Sentinel
- Sentinel: `SigninLogs` table
- Defender XDR: surfaced via the Entra ID connector in `AADSignInEventsBeta` (advanced hunting)
- Sysmon: not applicable (cloud identity source)
- Minimum retention recommended: 90 days

## Logic

The base rule (`name: entra_failed_signin_codes`) matches sign-ins whose `ResultType` is a known credential-failure code (50053 account locked, 50126 bad password, 50074 strong-auth required, and so on). The correlation rule references it by name and applies a `value_count` over `IpAddress`, firing when `gte: 10` distinct `UserPrincipalName` values are seen within `timespan: 1h`.

### SIEM-side aggregation

Sigma correlation rules require a backend that supports the correlation extension. For backends without native correlation support, port the threshold to a stateful query yourself:

- **Group key:** `IpAddress`
- **Counted field (distinct):** `UserPrincipalName`
- **Threshold:** ‚Č• 10 distinct users
- **Window:** 1 hour (sliding)

In Sentinel KQL this becomes a `summarize dcount(UserPrincipalName) by IpAddress, bin(TimeGenerated, 1h)` followed by `where dcount_UserPrincipalName >= 10`. In Splunk, `stats dc(UserPrincipalName) as users by IpAddress` inside a 1h time window with `where users>=10`. Convert with `sigma convert -t <backend> detections/sigma/T1110.003_entra-password-spray.yml` ‚ÄĒ backends that lack correlation support will emit the base rule and warn; apply the aggregation above manually there.

## False positives

- Misconfigured on-prem sync agents bombing Entra after a password change.
- Load-testing or vendor IAM tooling.
- NAT egress from a large office (many users behind one IP) hitting MFA after a policy change.

## Tuning

- Adjust the `gte` threshold by tenant size ‚ÄĒ small tenants 5, large tenants 20.
- Pivot on ASN instead of raw IP for distributed slow-and-low sprays from cloud providers.
- Enrich with `IdentityInfo` to prioritize sprays targeting admin roles.
