## Logic

Two YAML docs in one file:

1. **Base rule** (`outbound_https_connection`) â€” matches HTTPS GET/POST/PUT to port 443 with a small allowlist of well-known Microsoft / Office365 / Azure endpoints filtered out. Severity `low` because it fires on most legit traffic too.
2. **Correlation** â€” `event_count` grouped by `(id.orig_h, host)` over 1 hour with `gte: 30`. Severity `high`.

## SIEM-side aggregation note

The KQL sibling also enforces **rarity across the fleet** (â‰¤ 3 distinct devices have ever contacted the destination), which Sigma's current correlation primitives can't express directly. To get the same fidelity:

- **Splunk:** add `| stats dc(host) AS dc_devices ... by dest | where dc_devices <= 3` after the base search.
- **Elastic:** use an `aggs` query with `cardinality` on `source.ip` per `destination.domain`, then threshold.
- **Sentinel:** convert the Sigma to KQL then patch in `dcount(DeviceId)` (or just use the KQL sibling, which already does this).

## False positives

- Niche SaaS used by a single power user.
- Personal services on BYOD-tolerant networks.
- Auto-updaters for non-Microsoft software.

Tune the `filter_msft` block with your own well-known domains.

## Tuning

- Lower correlation `gte` to 10 and `timespan` to 30m for low-and-slow beacons.
- Add `useragent` / `user_agent` filter to drop browser-driven traffic if your Zeek deployment populates it.
- Cross-reference with TLS JA3 / JA4 fingerprint feeds (`ssl.log` in Zeek) for higher-fidelity attribution.
