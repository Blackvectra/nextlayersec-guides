## What it detects

Scheduled-task **creation or modification** from one of three command shapes, when the parent process is NOT a known installer:

- `schtasks.exe /create` / `/change` (and `-create` / `-change`).
- Legacy `at.exe \\<host>` network-task scheduler.
- PowerShell `New-ScheduledTask` / `Register-ScheduledTask` / `Set-ScheduledTask` cmdlets.

The `legit_installer_parent` selection excludes the common installer / orchestration parents (`msiexec`, `setup`, `install`, `update`, Google/Edge/Teams updaters, `OfficeClickToRun`, `TrustedInstaller`, `wuauclt`, `wusa`, SCCM `CcmExec`, Intune Management Extension, Chocolatey, winget). Anything else creating a scheduled task is suspicious.

The KQL companion (`detections/kql/T1053.005_scheduled-task-by-non-installer.md`) adds four boolean tag columns — writable-init / unsigned-init / LOLBin-sink — for richer triage; the Sigma rule keeps the entry condition portable.

## ATT&CK mapping

- **Tactic:** Persistence (TA0003) — second Sigma rule for this tactic (after T1547.001 Run keys).
- **Technique:** T1053.005 — Scheduled Task/Job: Scheduled Task
- **Related techniques:** T1059.001 (PowerShell — frequent task action), T1218.011 (Rundll32 — frequent task action), T1547.001 (Run keys — same malware often plants both).

## Data sources

- **Sigma logsource:** `product: windows`, `category: process_creation`.
- **Backends:**
  - **Defender XDR / Sentinel** — `sigma convert -t kusto`. The KQL companion has the per-pattern tags + scoring.
  - **Sysmon** — `sigma convert -t sysmon`. Maps to Event ID 1 (`ProcessCreate`).
  - **Splunk** — `sigma convert -t splunk`. Run against `index=wineventlog EventCode=4688` or Sysmon Event ID 1.
  - **Elastic** — `sigma convert -t elasticsearch`. Maps to `winlog.event_data.Image` / `winlog.event_data.CommandLine` / `winlog.event_data.ParentImage`.

## False positives

- In-house deployment / packaging tools — add to the `legit_installer_parent` selection in your fork.
- Vendor agents (printer / VPN / backup) registering service-maintenance tasks — verify signer before allowlisting.
- Developer / IT-automation workflows that register tasks from PowerShell — tune per environment.

## Tuning

- **Tighten by signer.** Backends with signer enrichment can add an exclusion for `ParentImage` paths under `\Program Files\` AND Microsoft-signed.
- **Extend the installer-parent allowlist** with your shop's deployment tooling — the upstream list is intentionally short and curated.
- **Raise priority** when the PowerShell branch matches AND the parent is not `msiexec.exe` / `chocolatey.exe` / `intune*`. PowerShell creating scheduled tasks from a non-installer chain is one of the highest-fidelity persistence signals on Windows.
