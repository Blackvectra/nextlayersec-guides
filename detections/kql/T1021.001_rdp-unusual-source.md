# RDP from unusual source

- **Query file:** `T1021.001_rdp-unusual-source.kql`
- **Author:** nextlayersec
- **Last updated:** 2026-06-08
- **Status:** testing
- **Severity:** Medium baseline; auto-raise to High on `B_External` or `B_SuccessAfterFails`.

## What it detects

Successful interactive RDP logons (`LogonType == "RemoteInteractive"`, `ActionType == "LogonSuccess"`) where the **source IP is not on the expected list**. Three independent tags:

| Tag | Meaning |
|---|---|
| `B_External` | Source IP is RFC-routable public space — not RFC1918, loopback, or CGNAT. RDP from the internet is almost never legitimate even when the firewall allows it; if VPN-less remote admin is your policy, you need to fix the policy. |
| `B_UnexpectedInt` | Source IP is internal but NOT in the per-environment `adminSubnets` or `jumpHosts` allowlist. Catches the lateral-movement pivot from a foothold workstation. |
| `B_SuccessAfterFails` | At least 5 failed RDP logons from the same source IP to the same target within the preceding 10 minutes. Classic password-spray-then-success signature. |

Three tags = three independent failure modes; the `Severity` column folds them into a SOC queue dial.

## ATT&CK mapping

- **Tactic:** Lateral Movement (TA0008) — first repo coverage for this tactic.
- **Primary technique:** T1021.001 — Remote Services: Remote Desktop Protocol
- **Co-fires with:** T1078 (Valid Accounts — stolen creds), T1110.001 / T1110.003 (Brute Force / Password Spray — drives `B_SuccessAfterFails`), T1133 (External Remote Services — `B_External`).
- **Related playbook:** [`blue-team-playbooks/incident-response-suspicious-network.md`](../../blue-team-playbooks/incident-response-suspicious-network.md), [`blue-team-playbooks/credential-theft-password-spray.md`](../../blue-team-playbooks/credential-theft-password-spray.md).

## Data sources

| Source | Table | Required event(s) |
|---|---|---|
| Defender for Endpoint | `DeviceLogonEvents` | `LogonType`, `ActionType`, `RemoteIP`, `DeviceName`, `AccountName` |

**Minimum connectors:** Defender for Endpoint Plan 1+. Runs natively in Defender XDR advanced hunting. In Sentinel, the same table is available via the **Microsoft Defender XDR** data connector.

**Windows EVTX equivalent:** map to Security log event ID **4624** (logon success) where `LogonType == 10` (RemoteInteractive). Use `IpAddress` field for source.

**Sysmon equivalent:** none directly — RDP-specific logging requires EVTX or MDE.

**Recommended retention:** 30 days (Defender XDR default).

## Logic

1. Scope to `LogonType == "RemoteInteractive"` and `ActionType == "LogonSuccess"` with non-empty `RemoteIP`.
2. Tag `B_External` using `ipv4_is_in_range` against RFC1918 / loopback / CGNAT.
3. Tag `B_UnexpectedInt` if source is internal AND not in `adminSubnets` AND not in `jumpHosts`.
4. Tag `B_SuccessAfterFails` via a left-outer join against the same table filtered to `LogonFailed` in the preceding 10-minute bucket per `(DeviceName, RemoteIP)` pair, counting ≥ 5.
5. Keep rows where any tag is set.
6. Score: `RiskScore = 3·External + 3·SuccessAfterFails + 2·UnexpectedInt`. `Severity` auto-raises to High on `External` or `SuccessAfterFails`.

## False positives

- **Site-to-site partner / vendor maintenance** — add the partner IP(s) to `jumpHosts` and document why.
- **Out-of-band admin** from an admin laptop on a roaming public IP — should be coming through VPN, but if your policy permits it, allowlist the source explicitly.
- **Penetration testing windows** — `B_External` will fire continuously during a Red Team. Pause the alert OR add the engagement window's source IP to a temporary allowlist with a calendar-style expiry comment in the KQL.
- **Mis-configured admin subnets** — if helpdesk reorganizes subnets and forgets to tell SOC, `B_UnexpectedInt` will fire. Run a monthly review of the `adminSubnets` allowlist against current network architecture.

## Tuning

- **`B_UnexpectedInt` precision** — the loudest branch in environments with many internal subnets. Replace the per-CIDR allowlist with a join on Microsoft Entra Connected Device IDs or with a custom watchlist of admin-tier device names.
- **`B_SuccessAfterFails` precision** — bump threshold from 5 to 10 if you have a noisy environment; tighten the time bucket from 10m to 5m for higher precision.
- **`B_External` precision** — if your policy genuinely allows internet-facing RDP for specific narrow use cases (don't), `B_External` will fire constantly. The right tuning is to fix the policy, not suppress the alert.
- **Add a username tier dial** — split the alert into two analytics: High-tier admin accounts (Domain Admins, Tier-0 service accounts) fire at any tag; regular users fire only on `B_External` or `B_SuccessAfterFails`.

## Output-data sensitivity

The `project` clause exports `AccountName` + `AccountDomain` — both directly identify users. The `RemoteDeviceName` exports a hostname that may identify endpoints. These are fine for SOC triage but should be access-controlled if alerts forward to broader audiences (e.g., ChatOps).

## Response

1. **Validate** — was this a known maintenance window, or unauthorized? Check change management.
2. **Pivot on the account** — query the last 24 hours of `IdentityLogonEvents` and `DeviceLogonEvents` for the same `AccountName` — any other sessions you didn't expect? Any sign-ins from anomalous IPs?
3. **Pivot on the source IP** — query the last 24 hours for any other RDP success or failure from this source.
4. **Snapshot the target** — if confirmed unauthorized, isolate the target via Defender for Endpoint **Isolate device**.
5. **Revoke sessions** — for the affected `AccountName`, revoke active sessions via Entra ID and force password reset + re-MFA-enrollment.
6. **Hunt fleet-wide** — for the same `RemoteIP` and `AccountName` across all `DeviceLogonEvents` since the earliest failure timestamp.
7. **Record** the case; cross-link any `T1110.003 password-spray` detections from the same source IP / account.
