# Entra Risky Sign-in + Mailbox-Rule Creation (post-AiTM)

- **Query file:** `T1078.004_risky-signin-mailbox-rule.kql`
- **Author:** nextlayersec
- **Last updated:** 2026-06-04
- **Status:** testing
- **Severity:** High — escalate to Critical when the rule forwards or redirects externally.

## What it detects

The single highest-fidelity signal that a **post-AiTM mailbox takeover** is in progress. After an attacker rides a stolen session cookie into M365, their first move is almost always to create an inbox rule that hides replies from the user (so the victim doesn't notice the wire-fraud or OAuth-consent emails the attacker sends). This rule joins a **risky-but-successful Entra sign-in** to a **`New-InboxRule` / `Set-InboxRule` / `UpdateInboxRules` / `Enable-InboxRule` event** from the same user within 10 minutes.

## ATT&CK mapping

- **Tactic:** Initial Access / Defense Evasion / Collection
- **Primary technique:** T1078.004 – Valid Accounts: Cloud Accounts
- **Co-fires with:** T1564.008 – Email Hiding Rules; T1098.005 – Account Manipulation: Device Registration (later in the kill chain)
- **Upstream:** see [`threat-intelligence/ttps/aitm-phishing-kits.md`](../../threat-intelligence/ttps/aitm-phishing-kits.md) for the kill chain that produces this signal.

## Data sources

| Source | Table | Required event(s) |
|---|---|---|
| Entra ID | `SigninLogs` | All interactive sign-in events with `RiskLevelDuringSignIn` / `RiskLevelAggregated` populated |
| Exchange Online (UAL) | `OfficeActivity` | `New-InboxRule`, `Set-InboxRule`, `UpdateInboxRules`, `Enable-InboxRule` |

**Minimum connectors:**

- Entra ID Diagnostic Settings → Log Analytics, with `SignInLogs` enabled.
- Office 365 Unified Audit Log → Log Analytics (Sentinel data connector "Office 365").
- Entra ID **P2** (or risk-based licensing) is required for `RiskLevelDuringSignIn` to be populated; the rule still works on P1 if you remove the `RiskLevel*` filters but FP rate goes up.

**Recommended retention:** 90 days (matches Sentinel default for `SigninLogs`).

## Logic

1. **Risky successful sign-in window** — pull all `ResultType == 0` sign-ins in the last 24h where Entra rated the sign-in or aggregated risk as `medium` or `high`, OR `RiskState` is non-empty.
2. **Mailbox-rule mutation window** — pull all `New-InboxRule` / `Set-InboxRule` / `UpdateInboxRules` / `Enable-InboxRule` events in the same window.
3. **Join on user + 10-minute time correlation** — same UPN, rule fires within 0–600 seconds of the sign-in.
4. **Extract high-signal rule attributes** — `MoveToFolder`, `MarkAsRead`, `DeleteMessage`, `ForwardTo`, `RedirectTo`. Any non-empty value on the last three is a near-certain compromise.

## False positives

- **Power users moving themselves into a new folder structure** right after sign-in (rare but happens).
- **Onboarding scripts / managed-mailbox tooling** that creates standardized rules immediately after a new device sign-in. Allowlist the responsible service account.
- **Legit autoresponders or vacation-rule changes** scheduled to coincide with sign-in.

Tune by:

- Suppressing the rule when `Operation == "Set-InboxRule"` AND no `MoveToFolder` / `ForwardTo` / `RedirectTo` / `DeleteMessage` parameter is set. (Pure flag changes are usually benign.)
- Adding a `where SigninDeviceCompliant != "true"` clause — compromised sessions almost always come from non-compliant devices.
- Joining `IdentityInfo` and excluding licensed automation accounts.

## Tuning

- **Tighten window** to 5m for higher fidelity (lower volume).
- **Widen window** to 30m if your tenant has high-latency log ingestion.
- **Add a positive-fidelity branch** for `ForwardTo` / `RedirectTo` set to an external domain — promote to Critical and page on-call.
- **Cross-reference** the `SigninIP` against threat-intel; an exact match on a known AiTM landing-page IP collapses this rule to "block now".

## Response

This is a **page-the-on-call** alert when the rule does any of: forward/redirect externally, delete, or move to RSS Feeds / Conversation History / Sync Issues. Actions:

1. **Revoke all sign-in sessions** for the user: Graph `revokeSignInSessions` or `Revoke-AzureADUserAllRefreshToken`.
2. **Reset password + force phishing-resistant MFA re-enrollment**.
3. **Disable + delete the inbox rule** via Graph or `Set-InboxRule -Enabled $false`.
4. **Hunt for OAuth consent grants** the user added during the compromise window (`Get-MgUserOauth2PermissionGrant`).
5. **Check for new device registrations** by that user in Entra.
6. **Pivot on `SigninIP` / `RuleClientIP`** — same IP touching other users in the tenant is a campaign indicator.
