# Beacon-like Outbound HTTPS to Rare Destination

- **Query file:** `T1071.001_beaconing-rare-https.kql`
- **Author:** nextlayersec
- **Last updated:** 2026-06-02
- **Status:** testing
- **Severity:** High when a workstation talks to a 1-device destination for > 1h; Medium otherwise.

## What it detects

Workstations sustaining outbound HTTPS to a destination that is **rare across the fleet** (low device fan-in). Most enterprise traffic concentrates on a small set of well-known endpoints (Microsoft, Google, vendor SaaS); a host steadily beaconing to a destination only it and 1–2 others have ever contacted is a high-fidelity C2 signal.

Catches: Cobalt Strike default Malleable C2 profiles, Sliver, Mythic, Brute Ratel, and most commodity loaders that beacon over HTTPS to attacker-controlled infrastructure.

## ATT&CK mapping

- **Tactic:** Command and Control (TA0011)
- **Technique:** T1071.001 – Application Layer Protocol: Web Protocols
- **Related:** T1573.002 – Encrypted Channel: Asymmetric Cryptography (the TLS layer)

## Data sources

- Table: `DeviceNetworkEvents` (Defender XDR / MDE)
- Equivalent for Sentinel-only: any HTTPS proxy / firewall / ZScaler / Defender for Cloud Apps logs with a `remote_url`, `device_id`, and `timestamp`.
- Retention recommendation: 30 days minimum, 90 days preferred — beaconing signal strengthens with a longer baseline.

## Logic

1. Filters successful or attempted **outbound** HTTPS connections (`RemotePort == 443`, public IPs only).
2. Excludes a rough allowlist of Microsoft / Office365 / Azure / `windowsupdate` endpoints. **You will need to expand this for your environment.**
3. Aggregates by `RemoteUrl + RemoteIP` and counts distinct devices that contacted it.
4. Keeps only destinations with **≤ 3 distinct devices** in the fleet AND **≥ 30 connection events** over a window of **≥ 30 minutes** at **≥ 0.5 connections / minute**.

The result is a destination that's *both rare and persistently contacted* — the defining shape of an interactive beacon. Returns the device sample and the process initiating the traffic for fast triage.

## False positives

- **Niche SaaS** used by one or two power users (developer tools, niche compliance vendors). Add to the allowlist by domain.
- **Personal services** on BYOD-tolerant networks (one user's iCloud, Dropbox, Steam).
- **Auto-updaters for non-Microsoft software** with chatty heartbeats.
- **Internal jump boxes** that legitimately make rare external calls.

Tune by:

- Joining `DeviceInfo` and restricting the rule to user-facing workstations (filter on `DeviceType in ("Workstation","Laptop")`), excluding servers and jump hosts.
- Excluding well-known auto-updater process names (`GoogleUpdate.exe`, `EdgeUpdate.exe`, `OneDrive.exe`, signed Adobe / Slack / Zoom updaters) via `InitiatingProcessFileName`.
- Growing the SaaS allowlist over the first 2 weeks of deployment.

## Tuning

- **Lookback window** — for very low-and-slow beacons, lift `lookback` to 7d (assumes ≥ 30d retention) and lower `minConnections` proportionally.
- **Process pivot** — `InitiatingProcessFileName` is a high-signal column. A beacon from `powershell.exe`, `rundll32.exe`, `mshta.exe`, `wscript.exe`, or any binary running out of `\Users\*\AppData\*` is near-certain malicious; promote those to its own analytic at lower thresholds.
- **JA3 / JA4 fingerprint** — when available (`AdditionalFields` on `DeviceNetworkEvents` for some product versions, or Zeek/Suricata feed), join on TLS fingerprint to catch beaconing tools regardless of destination rotation.
- **Reputation enrichment** — left-join with `ThreatIntelligenceIndicator` on `RemoteIP` / `RemoteUrl`; an exact match collapses this rule to "block now."

## Response

1. Isolate the host via EDR.
2. Pivot on `InitiatingProcessId` → process tree → parent. Identify the initial-access vector.
3. Snapshot memory + Run-key persistence (see [`T1547.001_runkey-persistence`](T1547.001_runkey-persistence.md)) before reimaging.
4. Block `RemoteUrl` + `RemoteIP` at the edge proxy / firewall and sinkhole DNS.
5. Hunt for the same destination across the fleet (`DeviceNetworkEvents | where RemoteUrl == "..."`).
