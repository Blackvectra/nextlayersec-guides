# Delivered Phish: Attachment + Credential-Harvester Link on Abused Hosting

- **Query file:** `T1566.001_attachment-link-credential-harvester.kql`
- **Author:** nextlayersec
- **Last updated:** 2026-06-04
- **Status:** testing
- **Severity:** Medium — raise to High on any `IsClickedThrough == true` (see clicker check).

## What it detects

The canonical **attachment-borne credential phish** that slips past filtering. An
inbound email is **delivered** (not blocked/quarantined), carries an
**attachment**, and contains an embedded URL pointing to a **credential
harvester hosted on abused free / trusted-SaaS infrastructure** (Google Sites,
Firebase, Azure Blob, `*.pages.dev`, etc.). The query weights the verdict with
**weak / best-guess sender authentication** — `SPF=none/fail` or
`DMARC=bestguesspass/none/fail` — because lookalike domains that pass DKIM but
have no real DMARC record are exactly what defeats reputation checks.

A representative pattern: a password-protected PDF from a DKIM-aligned lookalike
domain, carrying a `sites.google.com/view/...` harvester embedded as link
annotations, where SPF is `none` and DMARC is `bestguesspass`.

## Data sources

| Source | Table | Required event(s) |
|---|---|---|
| Defender for Office 365 | `EmailEvents` | Inbound mail with `DeliveryAction`, `AuthenticationDetails`, `AttachmentCount` |
| Defender for Office 365 | `EmailUrlInfo` | URLs extracted from message body / attachments |
| Defender for Office 365 | `EmailAttachmentInfo` | Attachment `FileName` / `FileType` / `SHA256` |
| Defender for Office 365 | `UrlClickEvents` | Safe Links click verdicts (clicker check) |

**Minimum connectors:**

- Microsoft Defender for Office 365 Plan 1+ (for `EmailEvents` / `EmailUrlInfo` /
  `EmailAttachmentInfo`). `UrlClickEvents` requires **Safe Links** (Plan 1+).
- Runs natively in Defender XDR advanced hunting. In Sentinel, the same tables
  are available once the **Microsoft Defender XDR** data connector is enabled.

**Recommended retention:** 30 days (Defender XDR advanced-hunting default).

## Logic

1. **Delivered inbound + attachment** — `EmailEvents` where
   `EmailDirection == "Inbound"`, `DeliveryAction == "Delivered"`,
   `AttachmentCount > 0`. Delivered = it bypassed filtering, which is the control
   gap we care about.
2. **Parse sender auth** — pull `SPF` / `DKIM` / `DMARC` / `CompAuth` out of
   `AuthenticationDetails` and compute `WeakAuth` (`bestguesspass` counts as weak,
   not a pass).
3. **Harvester URL join** — inner-join `EmailUrlInfo` on `NetworkMessageId` where
   `UrlDomain` matches the **abused-hosting list**.
4. **Attachment enrichment** — left-join `EmailAttachmentInfo` for filenames,
   types, and SHA-256s.
5. **Score + filter** — `RiskScore` adds points for weak auth, a
   `sites.google.com` host (a commonly abused harvester pattern), and PDF/HTML
   attachments; keep rows with weak auth OR `RiskScore >= 2`.

## False positives

- **Legitimate senders using free hosting** — newsletters, contractors, or SaaS
  vendors who genuinely host files on Google Drive / Firebase / Azure Blob and
  whose domain happens to fail strict SPF/DMARC.
- **Marketing mail** with tracking redirects that land on `*.netlify.app` /
  `*.vercel.app`.
- **Internal tools** that mail PDFs containing links to a corporate Firebase /
  Blob endpoint.

Tune by:

- Pruning `AbusedHosting` to only platforms you do **not** use legitimately, and
  allowlisting known-good `SenderFromDomain` values.
- Adding `| where DeliveryLocation != "Junk"` if you only care about inbox-landed
  mail.
- Requiring `WeakAuth == true` (drop the `RiskScore >= 2` branch) for a
  lower-volume, higher-precision rule.

## Output-data sensitivity

The `project` clause exports `Subject` and `RecipientEmailAddress` to the alert.
Both are useful for triage (you usually need the subject to recognize a
campaign quickly) but they carry information that may not match every SOC's
RBAC:

- **`Subject`** can contain HR / legal / M&A / medical / financial-deal
  vocabulary that should not be visible to every analyst. Examples:
  `"Re: Q3 acquisition target shortlist"`, `"FW: STD test results"`,
  `"Action required — board comp package"`.
- **`RecipientEmailAddress`** identifies the person who was phished and
  combined with the subject can be a small privacy disclosure.

If your alert pipeline (SIEM, SOAR, ticketing system, ChatOps notifier)
forwards alerts to recipients without need-to-know controls — drop `Subject`
from the projection or hash it. The triage analyst can pull the raw subject
on demand from `EmailEvents` with proper authorization. A safe minimal
projection is `Timestamp, NetworkMessageId, RecipientEmailAddress, SenderFromAddress,
SenderMailFromDomain, DeliveryLocation, HarvesterUrl, AttachmentNames, RiskScore`.

## Tuning

- **Impersonation watchlist (recommended):** maintain a watchlist of your
  protected / partner brand domains and add a fuzzy/edit-distance branch to
  promote lookalike `SenderFromDomain` values straight to High. This closes the
  common gap where anti-phishing impersonation protection does not catch a
  lookalike domain.
- **Encrypted-attachment branch:** password-protected attachments cannot be
  detonated. Add `AttachmentTypes has "pdf"` + a body-text match for a
  date-formatted passcode to flag the "passcode = today's date" trick.
- **Campaign pivot:** group by `SenderFromDomain` / harvester domain and alert
  when one indicator fans out across many recipients (multi-IP fan-out is a
  common tell).

## Response

1. **Purge** the message tenant-wide (Threat Explorer → soft/hard delete by
   sender domain).
2. **Block** the indicators: sender domain (TABL), harvester URL **path-level**
   (not the whole `sites.google.com`), attachment SHA-256.
   **Do not block** the sending IPs if they are shared Microsoft O365 frontends.
3. **Run the clicker check** (second query in the `.kql`). Any
   `IsClickedThrough == true` → revoke sessions, force password + MFA reset,
   hunt sign-ins for that user.
4. **Report** the harvester URL to Google Safe Browsing; **notify** the
   impersonated org.
5. **Record** the case in your internal incident tracker.
