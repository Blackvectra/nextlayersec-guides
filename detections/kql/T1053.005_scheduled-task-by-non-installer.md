# Scheduled task created or modified by a non-installer process

- **Query file:** `T1053.005_scheduled-task-by-non-installer.kql`
- **Author:** nextlayersec
- **Last updated:** 2026-06-11
- **Status:** testing
- **Severity:** Medium baseline; auto-raise to High when initiator is from a writable path, unsigned, OR the task action invokes a known LOLBin.

## What it detects

A scheduled task being created or changed by a process that is NOT on the known-installer allowlist. Four independent tag columns:

| Tag | Meaning |
|---|---|
| `B_NotInstaller` | Initiating process is NOT in the `knownInstallers` allowlist (`msiexec.exe`, `setup.exe`, Google/Edge updaters, SCCM `CcmExec.exe`, Intune Management Extension, Chocolatey, winget, etc.). Always set when the row is emitted (it's the entry condition). |
| `B_WritableInit` | Initiating process is running from a world-writable / user-controllable path (`\AppData\`, `\Public\`, `\ProgramData\`, `\Temp\`). Near-certain malware drop. |
| `B_UnsignedInit` | Initiating process has no signer info OR signer is "Unknown" / "Unsigned". |
| `B_LolbinSink` | The scheduled-task action invokes a known LOLBin — `powershell`, `mshta`, `rundll32`, `regsvr32`, `wmic`, `msbuild`, `cscript`, `wscript`, `installutil`, `msxsl`, `presentationhost`. The combination "task created by non-installer AND task runs a LOLBin" is one of the highest-fidelity persistence signals on Windows. |

Three command shapes covered:

- `schtasks.exe /create` / `/change` (and `-create` / `-change`).
- `at.exe \\<host>` (legacy network-task scheduler — should be near-zero in modern environments).
- PowerShell `New-ScheduledTask` / `Register-ScheduledTask` / `Set-ScheduledTask` cmdlets.

## Data sources

| Source | Table | Required event(s) |
|---|---|---|
| Defender for Endpoint | `DeviceProcessEvents` | `FileName`, `ProcessCommandLine`, `InitiatingProcessFileName`, `InitiatingProcessFolderPath`, `InitiatingProcessCommandLine`, `InitiatingProcessSignerType` |

**Minimum connectors:** Defender for Endpoint Plan 1+ (`DeviceProcessEvents` is in every plan). Runs natively in Defender XDR advanced hunting. In Sentinel, the same table is available once the **Microsoft Defender XDR** data connector is enabled.

**Sysmon equivalent:** Event ID 1 (`ProcessCreate`) with the same fields under different names (`Image`, `CommandLine`, `ParentImage`, `ParentCommandLine`, `IntegrityLevel`).

**Windows EVTX fallback (no MDE / Sysmon):** Security log event IDs:

- `4698` — A scheduled task was created.
- `4702` — A scheduled task was updated.
- `4699` — A scheduled task was deleted (use as a tamper-detection partner).

Map this rule to a watchdog query on those event IDs filtered by `SubjectUserName != SYSTEM`.

**Recommended retention:** 30 days minimum; 90 days strongly preferred for persistence detection (attackers plant tasks that don't fire for weeks).

## Logic

1. Scope to `DeviceProcessEvents` rows where the foreground process is one of `schtasks.exe`, `at.exe`, or PowerShell **with** a `*-ScheduledTask*` cmdlet in the command line.
2. Tag `B_NotInstaller` if `InitiatingProcessFileName` is NOT on the `knownInstallers` allowlist. Drop rows where it IS.
3. Tag `B_WritableInit` if the initiating process's folder path matches any world-writable root.
4. Tag `B_UnsignedInit` if `InitiatingProcessSignerType` is empty / "Unknown" / "Unsigned".
5. Tag `B_LolbinSink` if the task action (extracted from `/tr` / `--task-run` / `TaskRun` / `/action` / `/exec` argument) points at a known LOLBin.
6. Score: `RiskScore = 3·WritableInit + 2·UnsignedInit + 2·LolbinSink + NotInstaller`. `Severity` auto-raises to High on `WritableInit`, `UnsignedInit`, or `LolbinSink`.

## False positives

- **In-house deployment / packaging tools** (Chocolatey, Intune scripts, custom MSI wrappers). The `knownInstallers` list covers the common ones; add your shop's deployment tools to it.
- **Vendor agents that legitimately install scheduled tasks for service maintenance** (printer / VPN clients, some backup agents). Add to `knownInstallers` after verifying the binary is signed and the parent path is in `\Program Files\`.
- **Developer tooling** — engineers may register scheduled tasks for personal build / sync workflows. Tune per environment.
- **Sysinternals `Autoruns`** — listing tasks does not fire this rule (it doesn't `/create` or `/change`), but `Autoruns` running with `-i` to install ETW providers does and would fire on the `B_NotInstaller` path. Allowlist via signer.

## Tuning

- **Switch from filename allowlist to signer allowlist.** Replace the `knownInstallers` image list with a join on `DeviceFileCertificateInfo` filtering by `IsTrusted == true` AND signer organization in your trusted-vendor list. Fewer false negatives, less maintenance.
- **Tighten the LOLBin sink list.** If your environment legitimately uses `powershell.exe` in many scheduled tasks (build pipelines, IT automation), narrow `B_LolbinSink` to `mshta`, `rundll32`, `regsvr32`, `wmic` only.
- **Add a high-fidelity branch for hidden tasks.** Tasks created with `/RU SYSTEM` / `/IT` (Interactive Task hidden) / `\Microsoft\Windows\` paths intended to look legitimate are high-leverage to flag. Add a `B_HiddenName` tag matching task names that imitate the Microsoft tree (`\Microsoft\Windows\Anything-Not-On-Whitelist\`).
- **Volume tuning:** `B_NotInstaller` alone is the loudest. Surface only rows where `RiskScore >= 3` to the SOC queue, log lower scores to a hunting bucket.

## Output-data sensitivity

The `project` clause exports `InitiatingCmd` and `ScheduledTaskCmd` — both full process command lines. These can carry on-cmdline secrets from automation tooling (`--password=...`, `--token=...`). Trim or hash if your alert pipeline forwards to recipients without need-to-know controls.

## Response

1. **Snapshot the task definition.** Use Defender for Endpoint live-response (`getfile` on the task XML in `C:\Windows\System32\Tasks\<TaskName>`) or `schtasks /query /tn <TaskName> /xml` before any deletion. Evidence.
2. **Pivot on the initiating process tree.** Walk `InitiatingProcessParentFileName` and the surrounding `DeviceProcessEvents` for the same `DeviceName` ± 15 minutes — what spawned the process that created the task?
3. **Pivot on the task action.** What binary does the task actually run? Hunt for that binary fleet-wide (`DeviceFileEvents`, `DeviceImageLoadEvents`).
4. **Pivot on the user.** What other persistence has this user account touched in the last 30 days (Run keys, services, WMI subscriptions, startup folder)?
5. **Containment.** If confirmed malicious, isolate the device via Defender for Endpoint **Isolate device**. Delete the scheduled task via `schtasks /delete /tn <TaskName> /f` (or via the live-response shell).
6. **Hunt fleet-wide** for the same task name, action binary, and initiating process hash.
7. **Record** the case; cross-link to any Run-key persistence detections (T1547.001) from the same device.
